# 探索式测试（Exploratory Testing）

## 概述

探索式测试（ET）是一种同时进行测试设计、执行和学习的测试方法。与结构化测试（EQP/BVA/DCS/STT）不同，ET不预先设计所有测试点，而是依赖测试人员的经验、创造力和领域知识，在执行过程中逐步发现缺陷。

**本框架中ET的定位**：结构化测试的补充，而非替代。ET覆盖结构化测试难以触及的深层次、集成类和场景类缺陷。

## 适用场景

| 场景 | 说明 | 示例 |
|------|------|------|
| 复杂业务流程 | 多模块交互的端到端场景 | 从文件搜索→批量重命名→权限修改的完整命令链 |
| 历史缺陷高发区 | 之前迭代频繁出缺陷的模块 | 文件操作模块的路径处理和数据校验 |
| 外部系统集成 | 与外部系统的数据交互 | DNS解析结果与网络接口状态联动 |
| 数据完整性 | 数据在模块间流转的准确性 | 进程列表数据从采集到排序展示的传输 |
| 业务规则互斥 | 多个规则同时生效时的冲突 | 文件复制--checksum与--force的优先级冲突 |
| 文档模糊领域 | 需求未明确描述的操作组合 | 用户在非预期顺序下的命令参数组合 |
| 边界场景 | 正常流程之外的边缘操作 | 批量操作中途Ctrl+C中断、管道传输中断 |

## Session-Based Test Management (SBTM)

推荐使用基于会话的测试管理（SBTM）组织探索式测试。

### Charter（章程）

每个Session以明确的Charter为指引：

```
Charter格式：
探索 [目标区域] 使用 [关注点/资源] 发现 [目标]

示例：
探索 文件搜索功能 使用 特殊字符和通配符组合 发现 搜索逻辑的异常行为
探索 网络诊断模块 使用 网络中断和超时场景 发现 异常处理缺陷
探索 进程保护机制 使用 非标准操作顺序 发现 保护策略的绕过路径
```

### Time Box（时间盒）

| Session类型 | 建议时长 | 适用场景 |
|------------|---------|---------|
| 标准Session | 60分钟 | 复杂业务流程、多模块交互 |
| 快速Session | 30分钟 | 单一模块边界探索、快速检查 |
| 深度Session | 90分钟 | 高风险区域、历史缺陷重灾区 |

### Session结构

| 阶段 | 时长占比 | 活动 |
|------|---------|------|
| 章程定义 | 10% | 明确本次探索的目标、范围和关注点 |
| 探索执行 | 75% | 按Charter方向执行测试，记录发现 |
| 结果整理 | 15% | 整理发现的缺陷、问题和测试想法 |

## 探索时间配比建议

| 项目类型 | 建议ET占比 | 说明 |
|---------|-----------|------|
| 全新产品/大型迭代 | 15% | 需要大量探索理解产品行为 |
| 常规迭代 | 10% | 结构化测试为主，探索为辅 |
| 维护性迭代 | 5% | 最小化探索，聚焦回归 |
| 高风险模块（安全/支付/数据） | 20% | 加大探索投入降低风险 |

## 探索式测试与结构化测试的对比

| 维度 | 结构化测试（EQP/BVA/DCS/STT） | 探索式测试（ET） |
|------|---------------------------|-----------------|
| 目标 | 验证已知需求 | 发现未知问题 |
| 设计时间 | 执行前完成设计 | 设计执行同时进行 |
| 可重复性 | 高 | 低 |
| 发现缺陷类型 | 需求覆盖类缺陷 | 深层次/集成/场景类缺陷 |
| 依赖 | 需求文档 | 测试人员的领域知识 |
| 输出 | 可复用的测试点 | Session记录和缺陷报告 |

## 探索式测试的最佳实践

### 1. Charter设计原则
- **聚焦但不过窄**：Charter太宽则失去方向，太窄则失去探索意义
- **关注风险**：优先探索高风险区域
- **明确"发现"目标**：不要只写"探索登录功能"，要写"探索登录功能发现密码重置的安全漏洞"

### 2. 探索技巧
- **先正向后反向**：先走通正常流程，再思考"如果不这样会怎样"
- **数据变异性**：使用不同类型的数据（空值、超长、特殊字符、边界外）
- **操作顺序变异**：改变操作顺序（先搜索再过滤、先复制再修改源文件）
- **中断恢复**：在关键步骤中断操作（批量处理中Ctrl+C、断网后重连）
- **并发操作**：同时执行多个操作

### 3. 记录原则
- **即时记录**：发现异常立即记录，不要依赖记忆
- **区分观察和判断**：先记录"看到了什么"，再判断"是不是缺陷"
- **可复现性**：记录足够的上下文信息（数据、操作序列）以便复现
