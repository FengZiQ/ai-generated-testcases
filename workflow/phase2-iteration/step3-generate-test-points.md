# 阶段二·第三步：生成测试点

## 目标

基于变更分析和影响范围，生成本次迭代的测试点。

## 输入

1. 变更概览表（来自 Step1）
2. 影响范围分析表（来自 Step2）
3. 产品知识库中的功能点详情
4. `techniques/` 中的测试设计技术文件
5. **产品初始化时的行业分类和适用法规信息**（来自 `products/<product-name>/product-overview.md`）
6. **`reference/industry-regulations.md`**（行业法规参考）

## 处理流程

1. 从需求文档提取功能结构和验证点：
   - 识别需求文档中的功能区域/子模块划分
   - 识别每个功能区域下的输入参数、操作和输出
   - 提取每个验证点的约束信息（数据来源、值域范围、格式规则、依赖关系、业务规则等）
2. 对每个验证点，按"输入验证点生成规则"生成测试点：
   - 正常功能验证（输入有效参数/执行有效操作/检查预期输出）
   - 异常输入验证（无效值/空值/越界/格式错误等）
   - 边界条件验证（值域边界/长度边界/数量边界等）
   - 业务规则验证（组合条件/状态依赖/联动关系等）
   - **标注测试点的来源FP**：每个TP使用 `[来源: FP-MODxxx-xxx]` 后缀
3. **生成安全测试点（SEC）**：
   - 查阅 `techniques/security-testing.md` 评估每个模块的安全风险
   - 如果模块涉及：用户认证/授权、数据输入、文件上传、支付交易——必须生成 SEC 测试点
   - 每个模块至少覆盖：认证授权、注入防护、敏感数据保护三个领域
   - 跨模块安全测试点的 MODULE 使用 SYS
   - SEC 测试点标注风险等级
4. **生成性能测试点（PERF）**：
   - 查阅 `techniques/performance-testing.md`
   - 根据产品类型推断合理性能阈值
   - 识别性能敏感功能并生成 PERF 测试点
   - 至少覆盖：核心功能响应时间、典型并发量、稳定性
   - PERF 测试点标注预期性能阈值
5. **生成合规测试点（CMP）**：
   - 读取产品初始化时的行业分类和适用法规
   - 查阅 `reference/industry-regulations.md` 中对应行业的法规要求
   - 对每条适用的法规，生成 CMP 测试点
   - CMP 测试点标注引用的法规名称
6. 对影响范围内的回归功能点：
   a. 从影响范围分析表获取所有需要回归的功能点（FP）列表
   b. 对每个需要回归的FP，从产品知识库的模块文件中读取其描述和行为
   c. **为每个回归FP生成至少1个回归测试点**，覆盖该FP的核心行为是否因变更而受损
   d. 按优先级（P0/P1/P2）标注每个测试点，参考 `reference/quality-criteria.md` 中的"优先级分配规则"
   e. 在测试点描述尾部添加 `[回归]` 标签
   f. 在测试点统计表中新增优先级分布统计
7. **生成探索式测试建议**：
   a. 识别本次迭代中适合探索式测试的区域（复杂业务流程、历史缺陷高发区、外部系统集成、需求模糊领域）
   b. 参考 `techniques/exploratory-testing.md` 为每个识别区域生成ET Charter建议
   c. 规划ET Session数量和时间分配（占本次迭代测试总时间的10-15%）
   d. 输出ET Session建议表
8. **生成追溯矩阵**：
   a. 列出每个涉及的功能点（FP）及其对应的测试点（TP）列表
   b. **区分FP来源**：标注"新增FP"（本次迭代新增的功能点）和"回归FP"（本次迭代需要回归的已有功能点）
   c. 计算FP→TP覆盖率
   d. 标记未覆盖的功能点

## 测试点编写规范

1. **无步骤**：只描述验证点，不含操作步骤
2. **无具体测试数据**：描述数据条件，不用具体值
3. **预期结果可验证**：明确说明期望的行为或状态
4. **技术可追溯**：标注 EQP/BVA/DCS/STT

### 正确示例（按功能区域组织）

```markdown
#### 用户认证（FP-AUTH-001）
- AUTH-EQP-001：[P0] 验证有效邮箱格式+正确密码可成功登录 [来源: FP-AUTH-001]
- AUTH-EQP-002：[P0] 验证无效邮箱格式（缺少@）登录被拒绝并提示 [来源: FP-AUTH-001]

#### 密码安全（FP-AUTH-002）
- AUTH-EQP-003：[P0] 验证密码输入时掩码显示 [来源: FP-AUTH-002]
- AUTH-BVA-001：[P1] 验证密码长度边界值（7/8/64/65字符） [来源: FP-AUTH-002]

#### 用户注册（FP-AUTH-003）
- AUTH-EQP-004：[P1] 验证选填的出生日期可正常保存 [来源: FP-AUTH-003]
- AUTH-EQP-005：[P2] 验证出生日期留空时仍可完成注册 [来源: FP-AUTH-003]
```

### 错误示例
- ❌ "输入'abc@test.com'→输入密码→点击登录→验证登录成功"（含步骤和数据）
- ❌ "验证系统行为正常"（不明确）
- ❌ "测试登录功能"（太笼统）
- ❌ 按技术类型分组（违反了"按功能区域组织"原则，应将同一功能区域的测试点放在一起）

## 输入验证点生成规则

### 通用验证维度覆盖

对每个验证点，根据其类型（输入参数/操作/输出）至少覆盖以下维度：

| 维度 | 默认技术码 | 生成规则 | 默认优先级 |
|------|-----------|---------|-----------|
| 正常功能 | EQP | 输入有效参数/执行有效操作，验证输出正确 | P0 |
| 异常输入 | EQP | 输入无效值/空值/格式错误，验证被正确拒绝 | P0 |
| 边界条件 | BVA | 验证值域边界/长度边界/数量边界行为 | P1 |
| 业务规则 | DCS/STT | 组合条件/状态依赖/联动关系验证 | 按规则重要性 |

## 测试点 ID 分配

格式：`[MODULE]-[TECH]-[SEQ]`

- MODULE: 模块缩写（如 AUTH/ORDR/PAY/SYS，各产品独立定义）
- TECH: 测试设计技术码（EQP/BVA/DCS/STT/SEC/PERF/CMP）
- SEQ: 3位数字序号，每个模块+技术组合从001开始编号
- 从ID可识别所属模块和使用的测试设计技术

### 测试点标注规范

每个测试点必须包含以下标注：

1. **优先级标注**：在描述前加 `[P0]` `[P1]` `[P2]` 前缀，参考 `reference/quality-criteria.md` 的"优先级分配规则"
2. **来源追溯**：在描述后加 `[来源: FP-MODxxx-xxx]` 后缀，标注对应的功能点ID
3. **回归标识**：回归测试点在描述后加 `[回归]` 标签

示例：
```
AUTH-EQP-001：[P0] 验证有效邮箱格式被接受 [来源: FP-AUTH-001]
AUTH-BVA-001：[P1] 验证密码长度边界值 [来源: FP-AUTH-002] [回归]
```

## SEC 安全测试点生成规则

1. 对每个模块，评估其涉及的安全领域：
   - 用户数据输入 → XSS/注入测试
   - 认证/授权 → 越权/认证绕过测试
   - 文件上传/下载 → 文件安全测试
   - API/数据接口 → 访问控制/注入测试
   - 支付/交易 → 业务安全测试
2. 每个模块至少生成 2-3 个 SEC 测试点
3. 系统级安全测试点（跨模块）使用 MODULE=SYS
4. 每个测试点标注风险等级（高/中/低）
5. SEC测试点的风险等级与优先级对齐：高风险=P0、中风险=P1、低风险=P2

### 安全测试点示例
- SYS-SEC-001：验证所有需认证的接口在未登录状态被拒绝（高风险）
- AUTH-SEC-001：验证用户A无法通过修改参数查看用户B的数据（高风险）
- SRCH-SEC-001：验证搜索输入框的XSS攻击被转义（中风险）
- SYS-SEC-002：验证API响应不返回密码/令牌等敏感字段（中风险）

## PERF 性能测试点生成规则

1. 根据产品类型推断合理性能阈值（参考 `techniques/performance-testing.md`）
2. 识别性能敏感功能：列表查询/搜索/导出/推送/实时计算
3. 核心场景（P0）必须覆盖，标注具体性能预期值
4. 每个产品至少生成 3-5 个 PERF 测试点

### 性能测试点示例
- SYS-PERF-001：验证页面首屏加载时间不超过3秒（B端）/2秒（C端）
- ORDR-PERF-001：验证万级订单列表分页查询不超过1秒
- SRCH-PERF-001：验证组合条件搜索在3秒内返回结果
- SYS-PERF-002：验证200用户同时在线时系统稳定运行

## CMP 合规测试点生成规则

1. 从产品知识库获取行业分类和适用法规
2. 查阅 `reference/industry-regulations.md` 对应行业的法规细节
3. 对每条法规，提取可测试的合规要求
4. 转化为"验证xxx符合xxx法规"的测试点格式
5. 每个测试点标注引用的具体法规名称

### 合规测试点示例
- AUTH-CMP-001：验证用户注册时需明确同意隐私政策（个人信息保护法）
- PYMT-CMP-001：验证支付限额符合Ⅰ/Ⅱ/Ⅲ类账户规定（支付机构管理办法）
- SYS-CMP-001：验证UGC内容发布前经过敏感词审核（网络信息内容生态治理规定）
- USR-CMP-001：验证用户注销后个人数据在规定时间内清除（个人信息保护法）

### 安全测试点示例
- SYS-SEC-001：验证需认证接口在未登录状态被拒绝（高风险）
- AUTH-SEC-001：验证用户越权访问其他用户数据被拦截（高风险）

### 性能测试点示例
- SYS-PERF-001：验证页面首屏加载不超过[阈值]（核心场景）
- SYS-PERF-002：验证万级数据列表分页查询不超过[阈值]
