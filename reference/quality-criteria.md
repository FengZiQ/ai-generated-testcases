# 质量标准

## 测试点质量标准

| 标准 | 说明 | 严重等级 |
|------|------|---------|
| 覆盖率 | 每个需求至少对应一个测试点 | 高 |
| 技术覆盖率 | 每个识别出的技术都生成了测试点 | 高 |
| 预期结果明确 | 预期结果具体可验证 | 高 |
| 不含操作步骤 | 测试点不含"输入xxx→点击xxx"等操作描述 | 高 |
| 正反比例 | 正向:反向 ≈ 4:6 | 中 |
| 独立性 | 测试点之间无执行顺序依赖 | 高 |
| 命名规范 | [MODULE]-[TECH]-[SEQ] 格式正确 | 中 |
| 无模糊语言 | 不使用"应该正常""行为正确"等 | 中 |
| 粒度一致 | 同一模块内的测试点粒度保持一致 | 低 |

## 测试点优先级定义

| 优先级 | 定义 | 通过率要求 | 缺陷处理规则 |
|--------|------|-----------|-------------|
| P0 | 核心功能 / 关键业务流程 / 必须通过的路径 | 100% | 不允许存在未关闭的P0缺陷 |
| P1 | 重要功能 / 主要异常场景 / 高频使用功能 | >=95% | P1缺陷需确认不影响核心业务流程 |
| P2 | 边缘功能 / UI细节 / 次要异常 / 低频功能 | >=90% | 可协商延期，需记录已知问题清单 |

### 优先级分配规则

| 测试点类型 | 默认优先级 | 说明 |
|-----------|-----------|------|
| 新增功能的核心路径 | P0 | 迭代变更直接涉及的功能点 |
| SEC高风险 | P0 | 安全红线 |
| PERF P0场景 | P0 | 用户主流程性能 |
| 新增功能的异常场景 | P1 | 反向用例的主要部分 |
| SEC中风险 | P1 | 一般安全风险 |
| PERF P1场景 | P1 | 高频功能性能 |
| 边缘条件/UI验证 | P2 | 非关键路径 |
| SEC低风险 | P2 | 低影响安全 |
| 兼容性/易用性 | P2 | 非功能性补充 |
| 回归测试点 | 继承原测试点优先级 | 回归不改变优先级 |

## 回归策略

### 回归标识规范

采用**无后缀标记法**：不修改TP ID本身，仅在测试点描述尾部添加 `[回归]` 标签。

### 回归策略执行细则

| 影响等级 | 策略 | 选择规则 | 最低优先级覆盖 |
|---------|------|---------|--------------|
| 高 | 全量回归 | 变更模块的全部测试点 + 依赖链上所有模块的全部测试点 | P0+P1+P2 |
| 中 | 关键路径回归 | 变更模块的P0测试点 + 依赖模块的P0+P1测试点 | P0+P1 |
| 低 | 冒烟测试 | 变更模块的P0测试点 + 依赖模块的接口级P0测试点 | P0 |

## 测试点粒度分级标准

| 级别 | 名称 | 定义 | 覆盖范围 | 适用技术 |
|------|------|------|---------|---------|
| L1 | 功能级 | 验证一个完整功能或场景的整体行为 | 一个功能点的所有路径 | GEN, EQP(简单功能无分支) |
| L2 | 场景级 | 验证一个功能点在特定条件下的行为分支 | 一条特定条件路径 | EQP, DCS, STT, SEC, PERF, CMP |
| L3 | 条件级 | 验证单个输入条件的具体取值 | 一个具体字段/参数/状态值 | BVA |

### 粒度选择规则

| 技术类型 | 推荐粒度 | 说明 |
|---------|---------|------|
| EQP 等价类划分 | L2（场景级） | 每个有效/无效类是一个场景，不应细分到单值 |
| BVA 边界值分析 | L3（条件级） | 边界值天生需要具体数值验证 |
| DCS 决策表 | L2（场景级） | 一行决策规则是一个场景 |
| STT 状态转换 | L2（场景级） | 一个状态转换路径是一个场景 |
| SEC 安全测试 | L2（场景级）或L1（功能级） | 单攻击类型=场景级；全量扫描=功能级 |
| PERF 性能测试 | L2（场景级） | 一个性能场景 |
| CMP 合规测试 | L2（场景级） | 一条法规的一个可测试点 |
| GEN 通用功能 | L1（功能级） | 简单功能点确认 |

### 粒度一致性规则

1. **同一功能内**：同一功能的所有测试点应保持同一粒度级别
2. **同一模块内**：同一模块的不同功能可不同粒度，但测试点描述中应清晰标注
3. **同一技术类型内**：同一技术（如EQP）在模块内应保持粒度一致
4. **粒度降级原则**：不确定时优先选择L2（场景级）作为默认粒度

### 粒度一致性核查清单

- [ ] 同一功能内的EQP测试点是否粒度一致？
- [ ] 同一模块内的BVA测试点是否都是L3粒度？
- [ ] L1测试点是否描述了一个完整的、不可再分的功能？
- [ ] L3测试点是否过于琐碎？（单个字段验证是否可以合并？）
- [ ] 是否存在粒度跳级（L1和L3混在同个功能下）？如有，是否有合理理由？

## 完成标准

| 优先级 | 通过率要求 | 本次迭代目标 |
|--------|-----------|-------------|
| P0 | 100% | 本次迭代P0总数个，全部通过 |
| P1 | >=95% | 本次迭代P1总数个 |
| P2 | >=90% | 本次迭代P2总数个 |

### 退出条件

满足以下所有条件时，迭代测试可结束：
1. 所有P0测试点已执行且通过率100%
2. P1测试点通过率>=95%，且未通过的P1点已确认不影响核心流程
3. P2测试点通过率>=90%，且未通过的P2点已记录为已知问题
4. 无未关闭的P0缺陷
5. 探索式测试Session按计划完成（至少完成建议Session数的80%）
