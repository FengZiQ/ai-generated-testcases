# AI 生成测试用例框架

> 适用于敏捷开发团队的基于 Claude Code 的智能测试用例生成系统 —— 输入产品需求文档，输出结构化测试点清单和思维导图

## 项目概述

本项目是一套通用的、可复用的测试用例生成框架。它利用 Claude Code 的能力，将各种形态的产品需求文档（Wiki/Word/Excel/PDF/Markdown）转化为高质量的测试点文档，并支持输出 Markmap 思维导图。

### 核心理念

- **测试点优先**：只描述"测什么"，而非"怎么测"，聚焦验证内容
- **技术驱动**：系统化应用7种测试设计技术，而非凭经验随意编写
- **合规内置**：根据产品行业自动识别适用法规，生成合规测试点
- **两阶段流程**：新产品先初始化建立知识库，后续迭代基于变更增量生成
- **多产品管理**：一个项目实例可同时管理多个产品，各自独立初始化、独立迭代

### 适用角色

- 测试工程师：快速生成系统化的测试点，提高测试设计效率和质量
- 测试经理：统一团队测试设计标准，确保测试覆盖率，管理多条产品线
- 产品/开发人员：自测阶段快速生成测试checklist

## 快速开始

### 前提条件

- [Claude Code](https://claude.ai/code) 已安装
- 本项目的文件已就绪

### 场景一：初始化新产品

当你有一个全新的产品需要开始测试时：

```
> 帮我初始化新产品，需求文档在 xxx.docx
```

每个产品独立初始化，完成后与其他产品共存于 `products/` 目录下。Claude Code 会执行 **初始化三步流程**：

> 💡 **参照示例**：`products/example-mac-cli/` 目录下有一个完整的 Mac 命令行工具初始化示例，展示了初始化的输入需求和各产出物的格式。

1. 📖 **解析文档** — 读取并理解你的需求文档内容
2. 🧩 **提取模块** — 识别产品模块结构、功能点、行业分类
3. 🗺️ **构建知识库** — 生成产品概述和功能地图，保存到 `products/<产品名>/`

**初始化产出：**

| 产出物 | 存储位置 | 说明 |
|-------|---------|------|
| 产品概述 | `products/<产品名>/product-overview.md` | 产品基本信息、行业分类、适用法规 |
| 模块功能点 | `products/<产品名>/modules/` | 每个模块一个文件，含功能点详情 |
| 模块索引 | `products/<产品名>/modules-index.md` | 模块间依赖关系 |
| 风险评估 | 产品概述中 | 初始风险领域评估 |

### 场景二：迭代测试

产品初始化完成后，当有新迭代的需求变更时：

```
> 这是 xxx 产品 v1.1 的迭代需求文档，请生成测试点
```

Claude Code 会执行 **迭代测试四步流程**：

> 💡 **参照示例**：`products/example-mac-cli/iterations/v1.0/test-points.md` 是一个完整的迭代测试输出示例，包含 56 个测试点，覆盖全部 7 种测试技术。

1. 🔍 **理解变更** — 识别新增、修改、修复、配置变更
2. 📊 **分析影响** — 基于模块依赖关系分析回归影响范围
3. ✏️ **生成测试点** — 按7种测试技术生成功能+安全+性能+合规测试点
4. 🧠 **输出思维导图** — 生成 Markmap 兼容的思维导图文件

**迭代产出：**

| 产出物 | 存储位置 | 说明 |
|-------|---------|------|
| 测试点清单 | `products/<产品名>/iterations/<版本>/test-points.md` | 按模块和技术组织的测试点 |
| 思维导图 | `my-mind-maps/<产品名>-v<版本>-test-points.md` | Markmap 兼容格式 |
| 影响分析 | 迭代测试计划中 | 变更影响范围和回归建议 |

## 7种测试设计技术

| 技术码 | 技术名称 | 适用场景 | 示例测试点 |
|--------|---------|---------|-----------|
| **EQP** | 等价类划分 | 输入数据有有效/无效分类 | "验证无效邮箱格式（缺少@、缺少域名）被拒绝" |
| **BVA** | 边界值分析 | 数值范围或长度限制 | "验证密码长度边界值（7、8、64、65字符）" |
| **DCS** | 决策表 | 多条件组合的业务规则 | "验证邮箱有效×密码有效×凭证匹配×账户激活→登录成功" |
| **STT** | 状态转换 | 状态变化驱动的流程 | "验证连续5次失败后账户锁定，15分钟后自动解锁" |
| **SEC** | 安全测试 | 认证/授权/数据保护/注入防护 | "验证用户A无法通过修改参数查看用户B的数据（水平越权）" |
| **PERF** | 性能测试 | 响应时间/并发/稳定性 | "验证500用户同时登录时系统不崩溃，响应时间<3s" |
| **CMP** | 合规测试 | 行业法规和标准要求 | "验证用户注销后个人数据在规定时间内清除（个人信息保护法）" |

### 技术选择速查

| 需求特征 | 推荐技术 |
|---------|---------|
| 表单输入验证 | EQP（分类）+ BVA（边界） |
| 业务规则/权限控制 | DCS（条件组合）+ SEC（越权） |
| 审批流/订单流/工作流 | STT（状态转换） |
| 搜索/列表/导出 | EQP + BVA + PERF（大数据量性能） |
| 用户认证/登录 | BVA + DCS + STT + SEC（重点覆盖） |
| 支付/交易 | DCS + STT + SEC + CMP（金融法规） |
| 所有模块 | SEC（XSS/注入/越权/数据保护） |
| 所有产品 | PERF（响应时间/并发/稳定性） |
| 涉及用户数据 | CMP（个人信息保护相关法规） |

## 测试点 ID 命名规范

```
[MODULE] - [TECH] - [SEQ]
   │          │         └─ 3位数字序号（模块+技术组合内从001起）
   │          └─────────── 技术码：EQP/BVA/DCS/STT/SEC/PERF/CMP
   └────────────────────── 模块缩写：AUTH/ORDR/PAY/SYS等（各产品独立定义）
```

从ID可直接识别所属模块和使用的测试设计技术。

### 命名示例

- `AUTH-EQP-001`：认证模块、等价类划分、第1个
- `ORDR-BVA-001`：订单模块、边界值分析、第1个
- `PAY-STT-001`：支付模块、状态转换、第1个

## 测试点 vs 详细测试用例

本框架产出的是**测试点**而非**详细测试用例**，两者的区别：

| 对比维度 | 详细测试用例 | 测试点（本框架） |
|---------|-------------|----------------|
| 粒度 | 每一步操作都有描述 | 只描述验证内容 |
| 测试数据 | 含具体测试数据 | 描述数据条件即可 |
| 操作步骤 | 含具体操作步骤 | 无步骤 |
| 数量 | 多（一个场景拆多个用例） | 少（一个测试点覆盖一个场景） |
| 适用阶段 | 执行测试时 | 设计/评审时 |
| 维护成本 | 高（每次UI变更需修改） | 低（关注业务逻辑） |

### 示例对比

```diff
- ❌ 详细用例: "Step1:输入'user@example.com' → Step2:输入'Abc123!' → Step3:点击登录 → 预期:登录成功"
+ ✅ 测试点:    "验证有效邮箱+有效密码可成功登录"
+ ✅ 测试点:    "验证密码长度在8-64字符边界值（7、8、64、65）"
+ ✅ 测试点:    "验证SQL注入payload在登录接口被拦截"
```

### 性能测试
- SYS-PERF-001：[P0] 验证登录页首屏加载<2s [来源: FP-AUTH-001]

## 最佳实践

### 需求文档准备

- 按功能模块组织文档，每个模块有清晰的标题层级
- 明确标注数值约束（"密码长度为8-64字符"）
- 描述异常场景和错误处理
- 如果涉及合规要求，标注适用的法规名称

### 使用建议

- **首次使用**：先用一个熟悉的产品的简单模块试试，熟悉输出风格
- **大型产品**：建议按模块分批初始化，不要一次性处理整份超大文档
- **迭代频率高**：每次迭代都使用本框架生成测试点，保持知识库更新
- **结合手动设计**：框架生成的测试点是基础，测试工程师可在此基础上补充探索式测试点

### 常见问题

**Q: 需求文档是英文的怎么办？**
A: 框架不限制文档语言，中英文均可处理。

**Q: 产品涉及多个行业（如医疗电商）怎么办？**
A: 初始化时识别为多行业交叉，框架会合并多个行业的法规要求。

**Q: 生成的测试点太多/太少怎么办？**
A: 可以在提示Claude时指定"请聚焦核心功能"或"请尽量全面覆盖"。

**Q: 需求文档有歧义怎么处理？**
A: 框架会自动记录假设并继续生成，在输出中会标明假设内容供人工确认。

## 项目背景

本项目由资深测试工程师设计，目标是解决测试工作中以下痛点：

1. **测试设计效率低**：手动编写测试用例耗时且容易遗漏
2. **设计方法不统一**：不同测试人员的用例风格和深度差异大
3. **合规要求难覆盖**：行业法规更新快，容易遗漏合规测试点
4. **回归分析不准确**：迭代变更时难以准确评估影响范围
5. **跨团队协作难**：测试点的格式和标准不统一
